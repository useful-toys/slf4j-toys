# TDR-0034: Accepting Redundancy in fail(Throwable) Log Messages

**Status**: Accepted  
**Date**: 2026-02-10

## Context

When `Meter.fail(Throwable)` is called, the Meter now attaches the original `Throwable` to the SLF4J log event so that the logging framework can print the full stack trace. This produces output like:

```
01:00:07.873 [main] ERROR teste - FAIL: [java.lang.RuntimeException; Erro simulado] 47,2us; 18e83
java.lang.RuntimeException: Erro simulado
        at org.usefultoys.slf4j.Main.main(Main.java:23)
```

The exception type (`java.lang.RuntimeException`) and message (`Erro simulado`) appear twice:

1. In the human-readable message generated by `MeterDataFormatter`: `FAIL: [java.lang.RuntimeException; Erro simulado]`
2. In the stack trace printed by the logging framework: `java.lang.RuntimeException: Erro simulado`

The question is whether the readable message should be modified to omit the exception type and/or message when the `Throwable` is attached, since the stack trace already carries this information.

## Decision

Accept the redundancy and keep the readable message unchanged. The `failPath` continues to store the fully qualified exception class name and `failMessage` stores the exception message, exactly as before the `Throwable` attachment feature was introduced.

The readable message format `FAIL: [<failPath>; <failMessage>]` remains the same for all `fail()` variants, preserving consistency in how path identifiers are displayed across `ok()`, `reject()`, and `fail()` outcomes.

## Consequences

### Positive ✅

- **Consistency with other outcomes**: The `failPath` encoding follows the same `toPath()` pattern used by `ok(Throwable)` and `reject(Throwable)`, where the path is derived from the exception class name. No special case is introduced for `fail()`.
- **Resilience when stack traces are suppressed**: Many production logging configurations truncate or suppress stack traces (e.g., log aggregators, structured logging pipelines, or appender-level filters). The readable message retains the exception type and message as a self-contained summary even when the stack trace is not visible.
- **Zero test impact**: The `ReadableMessageLegacyTest` class has a large matrix of formatting scenarios. Introducing a variant of `readableMessage()` (whether via a boolean parameter or a separate method) would require duplicating or significantly expanding this test matrix. Accepting the redundancy avoids this cost entirely.
- **Simpler code**: No conditional logic is needed in `MeterDataFormatter` to detect whether a `Throwable` will be attached downstream. The formatter remains purely a function of `MeterData` fields, with no awareness of how the message will be consumed.

### Negative ❌

- **Visual redundancy**: Users see the exception type and message twice in console output when stack traces are enabled. This is cosmetically suboptimal but does not affect functionality or diagnostic value.

### Neutral ⚖️

- **DATA_FAIL unaffected**: The `DATA_FAIL` trace event (JSON5 format) always contains `failPath` and `failMessage` regardless of this decision, since it serves machine-parsable diagnostics.

## Alternatives Considered

### ❌ Variant of readableMessage() that omits cause details

**Description**: Create `readableMessage(boolean includeCause)` or a separate `readableMessageWithoutCause()` method that omits `failPath` and `failMessage` from the formatted string when the `Throwable` is attached. The `Meter.fail()` method would call the no-cause variant before passing the message to `messageLogger.error()`.

**Why rejected**: The `ReadableMessageLegacyTest` class already covers a large combinatorial space of formatting scenarios. Introducing a second variant would require duplicating most of these scenarios to validate the alternative output, significantly increasing test maintenance cost with minimal diagnostic benefit.

### ❌ Change MeterDataFormatter to omit failPath when failMessage is present

**Description**: Modify `MeterDataFormatter.readableStringBuilder()` to display only `[<failMessage>]` instead of `[<failPath>; <failMessage>]` when both are present, since the co-occurrence of `failPath` and `failMessage` only happens with `Throwable` causes.

**Why rejected**: This would change the existing readable message format for all `fail(Throwable)` scenarios, including the `DATA_FAIL`-independent readable output. It would also break the established pattern where the path identifier (the exception class name) is always visible in the readable message, making it inconsistent with how `ok(Throwable)` and `reject(Throwable)` display their paths.

### ❌ Do not store failPath/failMessage for Throwable causes

**Description**: When `fail(Throwable)` is called, skip storing `failPath` and `failMessage` since the `Throwable` is now attached to the log event and carries this information natively.

**Why rejected**: This would remove the exception type and message from the `DATA_FAIL` JSON5 trace event, which is used for machine-parsable diagnostics and does not carry the `Throwable` object. The `failPath` is also used by `MeterAnalysis.getPath()` for programmatic outcome inspection.
